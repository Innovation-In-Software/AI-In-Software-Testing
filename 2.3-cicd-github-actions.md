**Duration:** 45–60 minutes

## Goal

Wire an existing test suite into a GitHub Actions pipeline so that lint, tests, and coverage run automatically on every push and pull request.

By the end of this lab, you will be able to:

- Create a GitHub Actions workflow file using AI
- Run lint, tests, and coverage checks automatically on push
- Interpret a failing pipeline and trace it to its root cause
- Enforce a coverage threshold that blocks merges when coverage drops

---

## Scenario

Your team has been writing tests manually and with AI throughout this course. Now you need to make those tests automatic — so that no one can merge code that breaks a test or drops coverage below the team's agreed threshold.

## Starter Code

This lab reuses `discount.py` and the test files from Labs 1.1 and 1.3.

**If you are starting fresh**, create `discount.py`:

```python
def calculate_discounted_price(price, customer_type, coupon_code=None, is_holiday=False):
    if price < 0:
        raise ValueError("price cannot be negative")
    customer_type = customer_type.lower()
    if customer_type == "premium":
        price = price * 0.8
    elif customer_type == "standard":
        price = price * 0.9
    if coupon_code == "SAVE10":
        price = price * 0.9
    elif coupon_code == "SAVE20" and customer_type == "premium":
        price = price * 0.8
    if is_holiday:
        price = price * 0.95
    return round(price, 2)
```

And a minimal `test_discount.py`:

```python
from discount import calculate_discounted_price
import pytest

def test_premium_discount():
    assert calculate_discounted_price(100, "premium") == 80.00

def test_standard_discount():
    assert calculate_discounted_price(100, "standard") == 90.00

def test_negative_price_raises():
    with pytest.raises(ValueError):
        calculate_discounted_price(-10, "standard")

def test_save10_coupon():
    assert calculate_discounted_price(100, "standard", coupon_code="SAVE10") == 81.00

def test_holiday_discount():
    assert calculate_discounted_price(100, "guest", is_holiday=True) == 95.00
```

Create a `requirements.txt`:

```
pytest
pytest-cov
flake8
```

## Part 2: Ask AI to Generate the Workflow

Ask AI to write the GitHub Actions workflow file. 

### Suggested Prompt

```text
Write a GitHub Actions workflow file for a Python project.

The workflow should:
- Trigger on push to main and on pull requests targeting main
- Run on ubuntu-latest
- Use Python 3.11
- Install dependencies from requirements.txt
- Run flake8 to lint discount.py (max line length 100)
- Run pytest with coverage reporting
- Fail the build if line coverage drops below 80%

The source file is discount.py and tests are in test_discount.py.
```

Review the generated YAML before using it. Check:

- Does the trigger cover both `push` and `pull_request`?
- Is the Python version pinned?
- Does the coverage step use `--cov-fail-under`?

Create the workflow file at exactly this path (GitHub requires it):

```
.github/workflows/ci.yml
```

## Part 4: Break and Fix the Pipeline

You will break the pipeline four times, once per failure type. After each break, fix it and push again before moving to the next round.

---

### Round 1: Lint Failure

Add this constant to the bottom of `discount.py`:

```python
SUPPORTED_COUPON_CODES = {"SAVE10": "10% off for all customers", "SAVE20": "20% off exclusively for premium customers"}
```

Push:

```bash
git add .
git commit -m "introduce lint failure"
git push origin main
```

Go to the Actions tab. The **Lint** step should fail with an error like:

```
discount.py:18:101: E501 line too long (117 > 100 characters)
```

`E501` is flake8's line-length rule. The workflow sets `--max-line-length=100`, so any line over 100 characters fails the build. This is a realistic mistake — inline dictionaries and long strings are common offenders in real code reviews.

**Fix:** Break the dictionary across multiple lines:

```python
SUPPORTED_COUPON_CODES = {
    "SAVE10": "10% off for all customers",
    "SAVE20": "20% off exclusively for premium customers",
}
```

Push the fix, confirm the pipeline goes green before continuing.

---

### Round 2: Failing Test

Add this test to `test_discount.py`:

```python
def test_wrong_expected_value():
    assert calculate_discounted_price(100, "premium") == 999.99
```

Push:

```bash
git add .
git commit -m "introduce failing test"
git push origin main
```

The **Test** step should fail. The lint step passes because the code is syntactically valid. Only the test runner catches it.

**Fix:** Delete the broken test, push again, confirm green.

---

### Round 3: Coverage Drop

Delete three tests from `test_discount.py` so fewer branches are exercised. Remove the holiday, coupon, and guest tests.

Push:

```bash
git add .
git commit -m "reduce test coverage"
git push origin main
```

The **Test with coverage** step should fail with a message like `FAIL Required test coverage of 80% not reached`.

**Fix:** Restore the deleted tests, push again, confirm green.

---

### Round 4: Branch Coverage

Line coverage counts whether a line executed. Branch coverage also checks whether both sides of every `if` condition were exercised. Update the workflow to enforce branch coverage:

In `.github/workflows/ci.yml`, change the test step to:

```yaml
      - name: Test with coverage
        run: pytest test_discount.py --cov=discount --cov-branch --cov-fail-under=75 --cov-report=term-missing
```

Push this change:

```bash
git add .
git commit -m "enforce branch coverage"
git push origin main
```

The pipeline may fail even with all your tests in place because some branches in `discount.py` are not covered in both directions. Read the `term-missing` output to identify the uncovered branches.

**Fix:** Add a test that covers at least one missing branch, push again, confirm green.

## Part 5: Protect the Main Branch (Optional)

To prevent anyone from merging a failing PR:

1. Go to **Settings → Branches** in your GitHub repository
2. Add a branch protection rule for `main`
3. Enable **Require status checks to pass before merging**
4. Select your CI workflow as a required check

With this in place, a pull request with a failing test or low coverage cannot be merged without bypassing the protection manually.

## Key Takeaways

A CI pipeline does not add new tests — it makes the tests you already have impossible to bypass.

The three steps (lint → test → coverage) catch three different categories of problem:

- **Lint** catches style and syntax issues before the code even runs
- **Tests** catch behavioral regressions
- **Coverage threshold** catches tests that were deleted or never written

AI is useful for generating the workflow YAML, but the decisions behind it — what threshold, which branches, which checks are required — belong to the team.

A green pipeline is only as meaningful as the tests inside it.

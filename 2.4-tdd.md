# Day 2 Lab: Test-Driven Development with AI

## Lab 7: TDD with a GitHub Copilot Agent

**Duration:** 60–75 minutes
**Audience:** Software testers, QA professionals, and developers
**Tools:** Python 3, pytest, GitHub Copilot (agent), GitHub Actions

---

## Goal

Practice the red-green-refactor cycle using a GitHub Copilot agent as your developer. You write the failing tests. The agent implements the code. CI confirms the result.

By the end of this lab, you will be able to:

- Translate acceptance criteria into failing pytest tests
- Push a failing test suite to GitHub as a specification
- Trigger a Copilot agent via a PR comment to implement code
- Review and validate AI-generated implementation in a pull request

---

## Scenario

Your product team has submitted a new feature request. There is no code yet — just a user story and acceptance criteria. Your job is to write the tests before any implementation exists, then hand the specification to a Copilot agent to implement.

---

## User Story

> As a bank customer, I want to transfer money from one of my accounts to another, so that I can manage my funds without visiting a branch.

---

## Acceptance Criteria

1. The transfer amount must be greater than zero
2. The source account must have sufficient funds to cover the transfer
3. The source and destination accounts must be different
4. After a successful transfer, the source account balance decreases by the transfer amount
5. After a successful transfer, the destination account balance increases by the transfer amount
6. A successful transfer returns a transaction record containing: `from_id`, `to_id`, `amount`, `from_balance_after`, `to_balance_after`
7. Invalid inputs raise a `ValueError` with a descriptive message

---

## Starter Code

Create a GitHub repository named `tdd-bank-lab` and clone it locally.

Create `bank.py` with only this stub:

```python
def transfer_funds(from_account, to_account, amount):
    """
    Transfer amount from from_account to to_account.

    Each account is a dict: {"id": int, "balance": float, "owner": str}

    Returns a transaction dict on success.
    Raises ValueError for invalid inputs.
    """
    pass
```

Create an empty `test_bank.py` — you will fill this in Phase 1.

Create `requirements.txt`:

```
pytest
pytest-cov
```

Create `.github/workflows/ci.yml`:

```yaml
name: TDD CI

on:
  pull_request:
  workflow_dispatch:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -r requirements.txt
      - run: pytest test_bank.py -v
```

`pull_request` triggers the workflow automatically on every push to an open PR. `workflow_dispatch` lets you trigger it manually from the Actions tab before a PR exists.

Push everything to `main`:

```bash
git add .
git commit -m "add stub, empty tests, and CI workflow"
git push origin main
```

Go to the **Actions** tab in GitHub and run the workflow once manually to confirm the setup works before opening a PR.

---

## Phase 1: Red — Write the Failing Tests

Use AI to generate a test suite from the acceptance criteria. **Do not write or generate any implementation yet.**

### Suggested Prompt

```text
I am practicing test-driven development. I have a Python function stub and a set
of acceptance criteria. Write a complete pytest test suite for the function based
only on the acceptance criteria. Do not write the implementation.

Function stub:
[paste bank.py]

Acceptance criteria:
1. The transfer amount must be greater than zero
2. The source account must have sufficient funds
3. The source and destination accounts must be different
4. Source balance decreases by the transfer amount
5. Destination balance increases by the transfer amount
6. Returns a transaction dict with: from_id, to_id, amount, from_balance_after, to_balance_after
7. Invalid inputs raise ValueError with a descriptive message

Use clear test names. Create realistic account fixtures.
```

Copy the AI-generated tests into `test_bank.py`.

Before running, read each test. Ask yourself:

- Does this test correspond to a specific acceptance criterion?
- Does it test behavior (what the function should do) or implementation (how it does it)?
- Would this test still pass if the implementation were completely wrong?

Run the tests locally:

```bash
pytest test_bank.py -v
```

Every test should fail because `transfer_funds` returns `None`. If any tests pass, the test is not actually testing behavior — fix or remove it.

Record your red state:

| Total Tests | Passing | Failing |
|---|---|---|
| | 0 | all |

## Phase 2: Push the Red and Open a PR

Create a feature branch and push your failing tests:

```bash
git checkout -b feature/transfer-funds
git add bank.py test_bank.py
git commit -m "feat: add failing TDD tests for transfer_funds"
git push origin feature/transfer-funds
```

Open a pull request on GitHub from `feature/transfer-funds` to `main`.

CI will run automatically and fail — every test is red. This is intentional. The failing CI run is your specification. The agent will read the test output to understand what it needs to build.

---

## Phase 3: Trigger the Copilot Agent

On the pull request, post a comment to assign the implementation to the Copilot agent:

```
@copilot implement the transfer_funds function in bank.py so that all
tests in test_bank.py pass.

Rules:
- Do not modify test_bank.py
- Each account is a dict with keys: id, balance, owner
- The function must mutate account balances in place
- Raise ValueError (not Exception) for all invalid inputs
```

The agent will:

1. Read `test_bank.py` to understand the expected behavior
2. Write an implementation in `bank.py`
3. Push commits to your branch
4. CI will rerun automatically

Watch the CI status on the PR. When all checks go green, the agent has satisfied your specification.

---

## Phase 4: Review the Implementation

Go to the **Files Changed** tab in the PR. The agent should have only touched `bank.py`.

Review the diff carefully:

- Does the logic match all seven acceptance criteria?
- Are there any edge cases the tests do not cover but the implementation handles anyway?
- Are there any edge cases the tests require but the implementation handles incorrectly?

Pull the changes and run the tests locally to confirm:

```bash
git pull origin feature/transfer-funds
pytest test_bank.py -v
```

If any tests are still failing, post another comment on the PR describing the specific failure. The agent can iterate:

```
@copilot test_transfer_same_account is still failing. The function is not
raising ValueError when from_account and to_account have the same id.
```

Do not merge until all tests pass locally.

---

## Phase 5: Refactor via the Agent

Once CI is green, ask the agent to refactor the implementation for clarity without breaking the tests:

```
@copilot refactor bank.py for readability. Do not change the function
signature or behavior. All existing tests must still pass after the refactor.
Use descriptive variable names and avoid magic numbers.
```

Review the refactored diff in the PR. Run the tests one final time:

```bash
pytest test_bank.py -v
```

If all tests still pass, merge the PR.

---

## Part 6: Add a New Requirement

The product team has added one more acceptance criterion after the initial implementation:

> **Criterion 8:** Transfers must be rounded to two decimal places. A transfer of 10.999 should transfer 11.00.

1. Write a new test for this criterion yourself — do not use the agent yet
2. Push the test to the branch — CI will go red again
3. Post a comment asking the agent to handle the new criterion
4. Review the agent's change and confirm CI goes green

This round demonstrates the full TDD loop: requirement → red test → agent green → review.

## Discussion Questions

1. Did the AI-generated tests cover all seven acceptance criteria? Were there any gaps?
2. Did the agent touch any file other than `bank.py`? What would it mean if it had modified the tests?
3. When you reviewed the PR diff, did the agent's implementation match what you expected? Where did it surprise you?
4. What would have happened if you had written the implementation first and the tests second?
5. How does handing implementation to an agent change the role of the tester on a team?
6. At what point did you feel confident the implementation was correct — when the agent finished, or when CI went green?

## Key Takeaways

The failing test suite is the requirements document. When you push tests to a branch and hand them to an agent, you are not handing over control — you are handing over the typing. The specification was written by you, in the form of tests, before any code existed.

The agent produces code that passes the tests. Passing tests is not the same as correct code. A test suite that is incomplete or imprecise will produce an implementation that is incomplete or imprecise. The quality of the agent's output is bounded by the quality of the tests you wrote.

The human review step in Phase 4 is not optional. Read the diff. Run the tests locally. The agent does not know your business domain, your security requirements, or your team's code standards. You do.

**The tester's job in AI-assisted TDD is to write tests precise enough that no wrong implementation can pass them.**

# Day 2 Lab: Test-Driven Development with AI

## Lab 7: TDD — Tests First, Code Second

**Duration:** 60–75 minutes  
**Audience:** Software testers, QA professionals, and developers  
**Tools:** Python 3, pytest, ChatGPT or GitHub Copilot

---

## Goal

Practice the red-green-refactor cycle using AI as a collaborator — first to generate failing tests from a user story, then to generate an implementation that passes them.

By the end of this lab, you will be able to:

- Translate acceptance criteria into failing pytest tests
- Use AI to generate an implementation that satisfies a test suite
- Distinguish between tests that specify behavior and tests that verify implementation details
- Refactor code while keeping tests green

---

## Scenario

Your product team has submitted a new feature request. There is no code yet — just a user story and acceptance criteria. Your job is to write the tests before a single line of implementation exists, then use AI to generate the implementation that passes them.

---

## User Story

> As a bank customer, I want to transfer money from one of my accounts to another, so that I can manage my funds without visiting a branch.

---

## Acceptance Criteria

1. The transfer amount must be greater than zero
2. The source account must have sufficient funds to cover the transfer
3. The source and destination accounts must be different
4. After a successful transfer, the source account balance decreases by the transfer amount
5. After a successful transfer, the destination account balance increases by the transfer amount
6. A successful transfer returns a transaction record containing: `from_id`, `to_id`, `amount`, `from_balance_after`, `to_balance_after`
7. Invalid inputs raise a `ValueError` with a descriptive message

---

## Starter Code

Create a file named `bank.py` with only this:

```python
def transfer_funds(from_account, to_account, amount):
    """
    Transfer amount from from_account to to_account.

    Each account is a dict: {"id": int, "balance": float, "owner": str}

    Returns a transaction dict on success.
    Raises ValueError for invalid inputs.
    """
    pass
```

Create `test_bank.py` — you will fill this in during the lab.

---

## Why TDD Works This Way

In classical TDD, the developer writes tests before writing any code. Running the tests immediately produces failures (red) because the function does not exist yet. The developer then writes the minimum code needed to make the tests pass (green), and finally cleans up the code (refactor).

Using AI changes the speed of each phase, not the sequence. AI can generate a full test suite from acceptance criteria in seconds, and generate an implementation in seconds more. The discipline — tests first, code second — still matters. It forces you to think precisely about behavior before thinking about implementation.

---

## Phase 1: Red — Generate Failing Tests

Ask AI to write the test suite based on the acceptance criteria. **Do not write or generate any implementation yet.**

### Suggested Prompt

```text
I am practicing test-driven development. I have a Python function stub and a set
of acceptance criteria. Write a complete pytest test suite for the function based
only on the acceptance criteria. Do not write the implementation.

Function stub:
[paste bank.py]

Acceptance criteria:
1. The transfer amount must be greater than zero
2. The source account must have sufficient funds
3. The source and destination accounts must be different
4. Source balance decreases by the transfer amount
5. Destination balance increases by the transfer amount
6. Returns a transaction dict with: from_id, to_id, amount, from_balance_after, to_balance_after
7. Invalid inputs raise ValueError with a descriptive message

Use clear test names. Create realistic account fixtures.
```

Copy the AI-generated tests into `test_bank.py`.

Before running, read each test. Ask yourself:

- Does this test correspond to a specific acceptance criterion?
- Does it test behavior (what the function should do) or implementation (how it does it)?
- Is the expected value derived from the acceptance criteria or made up?

Now run the tests:

```bash
pytest test_bank.py -v
```

Every test should fail because `transfer_funds` returns `None`.

Record your red state:

| Total Tests | Passing | Failing |
|---|---|---|
| | 0 | all |

## Phase 2: Green — Generate the Implementation

Now ask AI to write the implementation that makes the tests pass. Share the tests — not just the acceptance criteria — so AI can see exactly what it needs to satisfy.

### Suggested Prompt

```text
Here is a pytest test suite for a Python function called transfer_funds.
Write the implementation of transfer_funds that makes all these tests pass.

Tests:
[paste test_bank.py]

Function signature and docstring:
[paste bank.py]

Requirements:
- Each account is a dict with keys: id, balance, owner
- The function should mutate the account balances in place
- Return a transaction dict with: from_id, to_id, amount, from_balance_after, to_balance_after
- Raise ValueError (not a generic Exception) for invalid inputs
```

Replace the `pass` in `bank.py` with the AI-generated implementation.

Run the tests:

```bash
pytest test_bank.py -v
```

Record your green state:

| Total Tests | Passing | Failing |
|---|---|---|
| | | |

If any tests are still failing, read the error message, identify the gap between the implementation and the test, and ask AI to fix the specific issue.

---

## Phase 3: Refactor — Improve Without Breaking

With all tests passing, clean up the implementation. The goal is to make the code clearer without changing its behavior. The tests are your safety net.

### Suggested Prompt

```text
Here is a working Python function. Refactor it for clarity and readability.

Rules:
- Do not change the function signature
- Do not change the behavior — all existing tests must still pass
- Use descriptive variable names
- Replace any magic numbers or unclear conditions with well-named variables or comments
- Keep it concise — do not add unnecessary abstractions

[paste bank.py]
```

Replace the implementation in `bank.py` with the refactored version.

Run the tests one more time to confirm nothing broke:

```bash
pytest test_bank.py -v
```

All tests must still pass. If any fail, the refactor changed behavior — ask AI to explain what changed and revert that specific part.

## Part 4: Stretch — Add a New Requirement

The product team has added one more acceptance criterion:

> **Criterion 8:** A transfer cannot leave the source account with a negative balance, even if the account had exactly the transfer amount available (i.e. a zero balance after transfer is acceptable, but going below zero is not).

This criterion may already be covered by your existing tests, or it may expose a gap.

1. Write a new test for this criterion manually — do not use AI yet
2. Run the test
3. If it fails, ask AI to fix just this case in the implementation without changing anything else
4. Re-run the full suite to confirm no regressions

## Extension: Let Copilot Agent Write the PR

This extension shows where AI-assisted TDD is heading. Instead of pasting tests into a chat window, you push the failing tests to GitHub and let a Copilot agent read them and open a pull request with the implementation.

### Setup

Make sure you have the CI workflow from Lab 6 (`2.3-cicd-github-actions.md`) active on your repo, or create a simple one that runs `pytest test_bank.py -v` on push. The failing tests need to be visible to the agent as a red CI run.

Push your failing tests (with `bank.py` still containing only `pass`):

```bash
git add bank.py test_bank.py
git commit -m "add failing TDD tests for transfer_funds"
git push origin main
```

Confirm CI fails on the push — the red run is the spec.

### Open Copilot Workspace

1. Go to your repository on GitHub
2. Open **Copilot Workspace** (the Copilot icon in the top bar, or `github.com/copilot/workspace`)
3. Describe the task:

```text
The tests in test_bank.py define a transfer_funds function for a banking application.
The function stub in bank.py currently returns None and all tests are failing.

Implement transfer_funds so that all tests in test_bank.py pass.
Do not modify the tests. Only modify bank.py.
```

4. Review the plan Copilot proposes before it writes any code
5. Let it generate the implementation
6. Review the diff — check that it only touched `bank.py` and that the logic matches your acceptance criteria
7. Click **Create Pull Request**

### Review and Merge

Open the PR on GitHub. The CI workflow will run automatically.

- If CI is green: review the implementation one more time, then merge
- If CI is red: read the failing test output, go back to Copilot Workspace, and describe what needs to change

After merging, pull the changes locally and run the tests yourself:

```bash
git pull origin main
pytest test_bank.py -v
```

### What Just Happened

Your failing tests were the entire specification. You did not describe the algorithm, the error handling, or the data structure to Copilot — you described the expected behavior through tests. Copilot read those tests and produced working code.

This is TDD at its most direct: **the test suite is the requirements document, and the AI is the developer.**

The step that remains human is the PR review. Copilot will produce code that passes the tests, but passing tests is not the same as correct code. A test suite that is incomplete or imprecise will produce an implementation that is incomplete or imprecise. The quality of the output is bounded by the quality of the tests.

---

## Key Takeaways

TDD is a discipline about the order of thinking, not just the order of writing code.

Writing tests from acceptance criteria forces you to ask: *what exactly should this function do, and how do I know it did it?* Those questions are harder to answer once the code already exists.

AI compresses the time cost of each phase, which can make TDD more practical for teams that previously avoided it because writing tests first felt slow. The risk is using AI to skip the thinking — generating tests and implementation in one shot without the red phase. That defeats the purpose.

The tests written in Phase 1 are only valuable if they were written without peeking at the implementation. If AI wrote the tests after seeing the code, they verify what the code does, not what it should do. That is a subtle but important difference.

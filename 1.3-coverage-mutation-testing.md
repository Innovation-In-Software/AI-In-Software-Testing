## Starter Code

This lab builds on `discount.py` from Lab 1. If you no longer have it, create it now:

```python
def calculate_discounted_price(price, customer_type, coupon_code=None, is_holiday=False):
    """
    Calculate the final discounted price for a customer.

    Business rules:
    - premium customers receive 20% off
    - standard customers receive 10% off
    - guests receive no customer discount
    - coupon code SAVE10 gives an additional 10% off
    - coupon code SAVE20 gives an additional 20% off, but only for premium customers
    - holiday promotion gives an additional 5% off
    - discounts are applied sequentially
    - final price is rounded to 2 decimal places
    - price must be greater than or equal to 0
    - unknown customer types are treated as guests
    - unknown coupon codes are ignored
    """

    if price < 0:
        raise ValueError("price cannot be negative")

    customer_type = customer_type.lower()

    if customer_type == "premium":
        price = price * 0.8
    elif customer_type == "standard":
        price = price * 0.9

    if coupon_code == "SAVE10":
        price = price * 0.9
    elif coupon_code == "SAVE20" and customer_type == "premium":
        price = price * 0.8

    if is_holiday:
        price = price * 0.95

    return round(price, 2)
```

Create `test_discount_partial.py` — this is your deliberately weak starting test file:

```python
from discount import calculate_discounted_price


def test_premium_discount():
    result = calculate_discounted_price(100, "premium")
    assert result == 80.00


def test_standard_discount():
    result = calculate_discounted_price(100, "standard")
    assert result == 90.00
```

---

## Why This Starting Point Works Well

Two passing tests is not the same as two tests that cover the code.

This starting file represents a common real-world pattern: a developer tested the most obvious inputs, all tests pass, and coverage is never measured. The result is a false sense of security.

This file leaves four executable lines completely untested:

- The `raise ValueError` line — a negative price is never passed
- The body of the `SAVE10` branch — the coupon is never applied
- The body of the `SAVE20` branch — the premium coupon is never applied
- The body of the `is_holiday` branch — the flag is always `False`

Branch coverage is also significantly lower than line coverage. Several `if` conditions are evaluated, but only as `False` — the `True` paths are never exercised.

**Branch coverage vs. line coverage:**

Line coverage counts whether a line was executed. Branch coverage counts whether every outcome of a condition was tested.

For example: `if is_holiday:` is executed every time the function runs, so the line is covered. But `is_holiday` is always `False` in our two tests. The `True` branch — `price = price * 0.95` — never runs. Branch coverage finds this gap; the coverage of the `if` line alone does not.

This is why branch coverage is the more meaningful metric.

## Part 6: Run Mutation Testing

Mutation testing deliberately introduces small bugs ("mutants") into your source code and checks whether your tests detect them. A mutant that your tests fail to detect "survives" — meaning your tests do not actually guard against that category of error.

Install mutmut:

```bash
pip install mutmut
```

Create a `setup.cfg` file in the same directory as `discount.py` to tell mutmut what to test:

```ini
[mutmut]
paths_to_mutate=discount.py
tests_dir=.
```

Run mutation testing:

```bash
mutmut run
```

This takes a minute or two. mutmut runs your full test suite once per mutation.

When it finishes, view the summary:

```bash
mutmut results
```

You will see output similar to:

```
discount.x_calculate_discounted_price__mutmut_5: survived
discount.x_calculate_discounted_price__mutmut_44: survived
```

View the exact change for a specific surviving mutant using the full name from the results:

```bash
mutmut show discount.x_calculate_discounted_price__mutmut_5
```

Example output:

```
--- discount.py
+++ discount.py
@@ @@
-    if price < 0:
+    if price <= 0:
```

This mutant changed `<` to `<=`. If no test catches it, your tests cannot distinguish between "reject negative prices" and "reject prices of zero or lower."

To view all surviving mutants at once:

```bash
mutmut show
```

## Common Insights Students Miss

- Passing tests with high line coverage does not mean tests are correct — it means the lines ran, not that the behavior was verified
- Branch coverage catches a different class of gap: conditions evaluated only in one direction
- A mutant that changes `< 0` to `<= 0` survives when there is no test for `price == 0`, even if zero is a valid documented input
- AI explains surviving mutants best when given the actual diff rather than a vague description
- Mutation testing is slow for large codebases — teams typically run it only on recently changed files
- A mutation score of 80–85% is considered strong in practice; pursuing 100% is usually not worthwhile

---

## Key Takeaways

Coverage metrics measure what your tests execute. Mutation testing measures whether your tests actually catch the kinds of bugs that occur in practice.

The two tools answer different questions:

- Coverage: *which lines and branches did the tests reach?*
- Mutation testing: *would the tests notice if the code were subtly wrong?*

AI is useful at both ends of this workflow:

- At the start: reading a coverage report and proposing targeted tests
- At the end: explaining a mutant diff and generating the specific test needed to kill it

Neither coverage nor mutation testing tells you whether you are testing the right things. That judgment still belongs to the tester. But together, they make it much harder to feel falsely confident in a weak test suite.

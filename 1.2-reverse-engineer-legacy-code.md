Use AI to understand and test a poorly documented function you did not write.

- Use AI to explain unfamiliar code without running it first

## Scenario

Your team has inherited a shipping cost calculator from a developer who left the company. There is no documentation, no tests, and no one remembers what the function is supposed to do.

Your job is to figure out what this function does, find any defects, generate a test suite, and fix the hidden bug.

```java
// ShippingCalculator.java
// v2 - updated rates Q3
public class ShippingCalculator {

    public static double calculateShipping(String dest, double wt, boolean xpr, String cust, int qty) {
        double b = 0;
        double r = 0;
        int z = 0;

        if (dest.equals("CA") || dest.equals("OR") || dest.equals("WA")) {
            z = 1;
        } else if (dest.equals("NY") || dest.equals("NJ") || dest.equals("CT")) {
            z = 2;
        } else if (dest.equals("TX") || dest.equals("FL") || dest.equals("GA")) {
            z = 3;
        } else {
            z = 4;
        }

        if (wt > 0 && wt <= 1.0) {
            b = 5.99;
        } else if (wt > 1.0 && wt <= 5.0) {
            b = 8.99;
        } else if (wt > 5.0 && wt <= 20.0) {
            b = 14.99;
        } else if (wt > 20.0) {
            b = 24.99;
        }

        switch (z) {
            case 1: r = 0.50; break;
            case 2: r = 0.85; break;
            case 3: r = 1.20; break;
            default: r = 2.00; break;
        }

        double total = b + (wt * r);

        if (xpr) {
            total *= 1.75;
        }

        if (cust != null) {
            if (cust.equals("gold")) {
                total *= 0.80;
            } else if (cust.equals("silver")) {
                total *= 0.90;
            } else if (cust.equals("bronze")) {
                total *= 0.95;
            }
        }

        if (qty > 1) {
            total += (qty - 1) * 0.50;
        }

        if (z > 2) {
            total += 3.75;
        }

        return Math.round(total * 100.0) / 100.0;
    }
}
```

## Why This Code Works Well for This Exercise

Legacy code in production systems often has several characteristics that make it hard to understand and test:

- Abbreviated variable names with no clear meaning (`wt`, `xpr`, `cust`, `b`, `r`, `z`)
- No method documentation or parameter descriptions
- Complex nested conditionals with hardcoded values
- Multiple business rules combined in a single function
- A comment that tells you nothing useful (`// v2 - updated rates Q3`)

This function has all of these traits. It also contains a hidden bug that is easy to miss during a code review but straightforward to detect through testing.

## Part 6: Find the Bug

If one of the AI-generated tests failed when passing a lowercase state code, you have found the hidden bug.

Trace through the code manually with the input `("ca", 2.0, false, null, 1)`:

| Step | Question | Your Answer |
|---|---|---|
| 1 | What zone does `"ca"` map to? | |
| 2 | What base cost `b` is assigned for 2.0 lbs? | |
| 3 | What per-pound rate `r` is assigned for that zone? | |
| 4 | What is `total` after `b + (wt * r)`? | |
| 5 | Is the fuel surcharge applied? | |
| 6 | What is the final return value? | |

Now repeat the trace with `("CA", 2.0, false, null, 1)` and compare the two results.

### The Fix

Add one line to normalize the destination before the zone comparison:

```java
dest = dest.trim().toUpperCase();
```

Insert it as the very first line inside the method, before the zone assignment block.

After fixing, re-run the tests:

```bash
mvn test
```

All previously-failing tests related to lowercase state codes should now pass.

## Common Missed Cases

Students often miss some of these cases during manual reading:

- `dest` comparison is case-sensitive — `"ca"` and `"CA"` produce very different results
- `wt` of exactly `0` falls through all weight branches and `b` remains `0`
- Negative `wt` is accepted without error — `b` stays `0` and the rate multiplies a negative number
- `cust` of `null` is handled, but `cust` of an unrecognized string like `"platinum"` is silently ignored
- `qty` of `0` or negative does not trigger the multi-item surcharge — but may not be a valid business input
- The fuel surcharge applies to zones 3 and 4, so zone 4 (the fallback for any unrecognized state, including all lowercase inputs) is always surcharged

## Hidden Bug Reveal

### The Bug

The function compares `dest` using `.equals()`, which is case-sensitive in Java.

If a caller passes `"ca"` instead of `"CA"`, none of the zone 1, 2, or 3 conditions match. Java falls through to the `else` branch and assigns `z = 4`.

Zone 4 gets the highest per-pound rate (`r = 2.00`) and also triggers the fuel surcharge because `z > 2` is true.

### The Impact

| Input | Zone | Rate | Fuel Surcharge | Final Cost |
|---|---|---|---|---|
| `"CA"`, 2.0 lb, no express, no discount | 1 | $0.50/lb | No | $9.99 |
| `"ca"`, 2.0 lb, no express, no discount | 4 | $2.00/lb | Yes (+$3.75) | $16.74 |

The same package shipped to the same destination costs **$6.75 more** when the state code is lowercase — a 68% overcharge.

### Why It Is Easy to Miss

- The comparisons look correct at first glance
- A manual reader tends to assume the calling code passes normalized input
- There is no documentation describing what format `dest` must be in
- The bug only surfaces when a different part of the system passes improperly formatted input

### Why AI Often Finds It

AI models recognize the pattern of `.equals()` comparisons on string inputs without prior normalization. This is a known category of input-handling defect. A well-prompted AI will flag it as: *"The function does not normalize `dest` before comparison, so lowercase or mixed-case inputs will fall through to zone 4 and be overcharged."*

## Key Takeaways

AI is a powerful ally for reverse engineering legacy code, but it requires good prompting and human verification.

When using AI to understand unfamiliar code:

- A zero-shot explanation prompt gives you a starting point, not the full picture
- Asking AI to find defects often surfaces issues that manual reading misses
- AI-generated tests are most useful when you direct them toward boundary values and edge cases
- The hidden bug in this lab represents a real class of defect found in production systems
- AI tools recognize common defect patterns because they have seen similar code thousands of times

Good testers still need to:

- Verify AI explanations against actual code behavior
- Run the tests rather than trusting the AI's predicted output
- Use failing tests as a starting point for root cause analysis
- Confirm the fix is correct before closing the issue

AI is most valuable when treated as a code comprehension and test generation assistant, not as the final authority on correctness.
